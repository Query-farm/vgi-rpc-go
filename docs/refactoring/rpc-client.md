# RPC Client Interfaces

## Purpose

This document defines the client-side contract for the `rpc` package only:

- public client API
- introspection and capability discovery
- unary and streaming invocation APIs
- response reader contracts
- cancellation and external payload transport rules

Server-side APIs are documented separately in `rpc-server.md`.

## Core Principles

- `rpc.Client` is a low-level transport client, not a catalog API.
- Client code should remain Arrow-first.
- The client should reuse the same core transport types where possible:
  - `Request`
  - `Metadata`
  - `ProtocolVersion`
  - `RequestIDFromContext(...)`
- HTTP remains the default remote transport, but the client model should avoid HTTP-specific objects in the public surface unless they are genuinely transport-specific.

## Client

```go
type Client struct { ... }

func NewClient(baseURL string, opts ...ClientOption) *Client

func (c *Client) Describe(ctx context.Context) (*DescribeInfo, error)
func (c *Client) DescribeMethod(ctx context.Context, name string) (*MethodInfo, error)
func (c *Client) Capabilities(ctx context.Context) (*ExternalBatchCapabilities, error)

func (c *Client) Unary(name string, req *Request) (*Response, error)
func (c *Client) Consume(name string, req *Request, consumer Consumer) error
func (c *Client) Exchange(name string, req *Request, consumer Consumer) error
func (c *Client) Cancel(ctx context.Context, req *Request) error
```

Notes:

- `DescribeMethod(...)` is a convenience helper over `Describe(...)`, not a separate wire endpoint.
- `Capabilities(...)` is a thin wrapper over HTTP capability discovery.
- `Cancel(...)` is best-effort and should be used when the client knows it is done with an active producer or exchange stream.
- The client should preserve request metadata, request IDs, and tracing metadata across calls.

## Client Options

```go
type ClientOption interface {
    applyClient(*Client)
}
```

Expected option categories:

- custom `http.Client`
- authentication
- tracing
- logging
- retry boundaries
- transport limits

Notes:

- Retry and circuit-breaking policy should not be hard-wired into core `rpc.Client`.
- The core package should allow transport hooks and middleware-like integration points without forcing a single resiliency strategy.

## Shared Request Model

The client should use the same `Request` transport envelope as the server-side core model.

Practical expectations:

- `Request.Context()` is the canonical per-call context.
- `Request.Reader()` provides the Arrow IPC payload carried by the current request envelope.
- `Request.RequestID` is the canonical correlation identifier when provided by the caller or generated by runtime.
- `Request.Metadata` may include transport keys such as:
  - `vgi_rpc.request_id`
  - `vgi_rpc.log_level`
  - `traceparent`
  - `tracestate`
- `Request.State` is the outgoing continuation state for streaming calls.

Client-side usage:

- when passed into `Unary(...)`, `Consume(...)`, `Exchange(...)`, or `Cancel(...)`, `Request` is the outbound transport envelope
- when passed into `Consumer.Consume(...)`, `Request` is the inbound continuation envelope for the current step

## Introspection Types

### `type MethodInfo`

```go
type MethodInfo struct {
    Name         string
    MethodType   string
    Doc          string
    HasReturn    bool
    HasHeader    bool
    IsExchange   *bool

    ParamsSchema *arrow.Schema
    InputSchema  *arrow.Schema
    OutputSchema *arrow.Schema
    HeaderSchema *arrow.Schema

    ParamTypes    string
    ParamDefaults string
    ParamDocs     string

    Metadata Metadata
}
```

Notes:

- This is a lossless client-side representation of one `__describe__` row.
- `ParamTypes`, `ParamDefaults`, and `ParamDocs` intentionally preserve wire-level JSON payloads.
- `InputSchema` and `OutputSchema` may be nil when the method is dynamic and finalized only during init.

### `type DescribeInfo`

```go
type DescribeInfo struct {
    ProtocolName    string
    ProtocolVersion string
    DescribeVersion string
    ServerID        string
    Methods         []MethodInfo
    Metadata        Metadata
}
```

Notes:

- This is the parsed result of `__describe__`.
- `ProtocolVersion` should match `ProtocolVersion` from the local package when the peer is compatible.

## Response Model

### `type Response`

```go
type Response struct {
    Header   arrow.RecordBatch
    Metadata Metadata
    State    []byte
    reader   arrow.RecordReader
    final    bool
}

func (r *Response) Reader() arrow.RecordReader
func (r *Response) IsFinal() bool
```

Notes:

- `Response` is the unified client-side result envelope for unary, init, and stream steps.
- `Reader()` exposes the Arrow payload for the current response.
- `State` is the continuation token for streaming responses.
- `IsFinal()` reports whether the current streaming response has a next step.
- For unary responses, `IsFinal()` is always true.
- Log and error batches are handled by runtime and do not appear in `Reader()`.

### `type Consumer`

```go
type Consumer interface {
    Init(ctx context.Context, resp *Response) error
    Consume(r *Request, w ResponseWriter) error
}
```

Notes:

- `Consumer` is the unified client-side streaming interface for both producer and exchange methods.
- The initial init request payload is provided by `Consume(...)` / `Exchange(...)` through their `req *Request` argument.
- `Init(...)` receives the parsed init response exactly once before continuation begins.
- `Consume(...)` receives the current incoming continuation request shape and prepares the current outgoing step through `ResponseWriter`.
- `ResponseWriter` is the same transport-level writer contract defined in `rpc-server.md`.

### Client-side `ResponseWriter` phase rules

Client-side streaming reuses the same `ResponseWriter` contract, but only a subset of operations is meaningful.

Phase expectations:

- `Consumer.Init(...)`
  - client does not use `ResponseWriter`
  - the init request is already provided by the outer `req *Request` argument to `Consume(...)` / `Exchange(...)`
- `Consumer.Consume(...)`
  - `arrow.RecordWriter.Write(...)`
    - valid
    - writes the outgoing continuation payload for the current step
  - `SetMetadata(...)`
    - valid
    - attaches transport metadata for the outgoing continuation request
  - `SetHeader(...)`
    - valid
    - attaches an optional typed outgoing continuation header
  - `NextState(...)`
    - valid
    - sets the outgoing continuation state token for the current step
  - `Finalize()`
    - valid
    - closes the local outgoing step payload
  - `SetOutputSchema(...)`
    - invalid on the client side
    - output schema declaration is a server-side concern

Rules:

- Invalid client-side `ResponseWriter` operations must return an error.
- Client-side runtime should document clearly that `ResponseWriter` is reused for symmetry, but its valid surface is narrower than on the server.
- Client-side `ResponseWriter` validity rules are independent from server-side phase-validity rules, even though the interface name is shared.

## Cancellation

Notes:

- `Cancel(...)` accepts a `Request` so the client can reuse the already known continuation state and metadata in the same transport envelope shape used by other client calls.
- `Cancel(...)` maps to `POST /{method}/cancel` in the HTTP binding.
- `req.State` is the canonical stream identity for best-effort cancellation.

## Streaming Runtime Shape

Rules:

- `Consume(name, req, consumer)` and `Exchange(name, req, consumer)` use `req` as the initial outbound init request.
- After the init round completes, runtime parses the init result and passes it to `consumer.Init(ctx, resp)`.
- Each subsequent continuation round is exposed to `consumer.Consume(r, w)`.
- The continuation loop is fully runtime-managed inside one top-level client call.
- User code does not construct follow-up requests explicitly.
- After the initial init call, every subsequent continuation round uses:
  - `POST /{method}/exchange`
- `consumer.Consume(r, w)` represents one active continuation step.
- `r` exposes the inbound side of the current step.
- `w` builds the outbound side of the same current step.
- The runtime may keep both sides of the step open at the same time while the HTTP exchange is in progress.
- Client code must therefore treat `r` and `w` as a duplex pair for one logical step, not as two different sequential requests.
- Client-side streaming remains stateless at the public API level:
  - continuation identity lives in `r.State`
  - outbound continuation state is set through `w.NextState(...)`

## External Payload Transport

The client must understand the same externalization rules as the server transport binding.

Client expectations:

- capability discovery may happen through `HEAD /`
- request upload negotiation uses `__upload_url__/init`
- upload target rows contain:
  - `upload_url`
  - `download_url`
  - `expires_at`
- pointer batches may carry:
  - `vgi_rpc.location`
  - `vgi_rpc.location.sha256`
  - `vgi_rpc.stream_state#b64`

Notes:

- Externalization is transport-managed.
- Public client APIs should continue to expose Arrow readers and parsed responses rather than raw pointer-batch mechanics wherever possible.

## Error and Log Handling

Rules:

- Protocol log batches should be handled by runtime and not mixed into user-facing data readers.
- Protocol error batches should be surfaced as ordinary Go `error` returns.
- HTTP `X-VGI-RPC-Error: true` remains a transport-level implementation detail of the HTTP binding.

## Open Questions

- Whether `Capabilities(...)` should return `ExternalBatchCapabilities` directly or a client-specific wrapper that preserves additional raw metadata.
- What exact runtime ordering should be guaranteed for client-side `Consume(...)` when the current step both consumes an incoming payload and writes the next outgoing payload.

## Client-side Audit

### Covered

- transport-level client API
- `Describe(...)` and `DescribeMethod(...)`
- HTTP capability discovery wrapper
- unary invocation through `Unary(...)`
- unified streaming surface through `Consumer`
- best-effort cancellation through `Cancel(...)`
- unified `Response` envelope for unary, init, and stream step results
- reuse of shared transport types:
  - `Request`
  - `Metadata`
  - `ProtocolVersion`
  - `RequestIDFromContext(...)`
- external payload transport awareness
- external batch storage interaction:
  - capability discovery through `HEAD /`
  - upload URL negotiation through `__upload_url__/init`
  - pointer batch handling through:
    - `vgi_rpc.location`
    - `vgi_rpc.location.sha256`
    - `vgi_rpc.stream_state#b64`
- protocol log and error handling rules

### Covered With Explicit Runtime Rules

- dual client-side role of `Request`:
  - outbound envelope for top-level client calls
  - inbound envelope inside `Consumer.Consume(...)`
- narrower client-side validity surface for `ResponseWriter`
- stateless continuation model at the public API level

### Residual Risks

- `Request` is intentionally dual-use, so implementation must keep inbound and outbound runtime construction strict.
- `ResponseWriter` is reused on the client side for symmetry, but only a subset of operations is valid there.
- The exact ordering guarantees for `Consumer.Consume(...)` remain to be fixed precisely during implementation.
